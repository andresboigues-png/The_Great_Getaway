# 4.8 audit MK1

A full-codebase audit of **The Great Getaway**, run to take the app "to the max" on Opus 4.8. Two areas, kept strictly separate as requested:

- **PART 1 — Bugs & use-case findings** (functional correctness, security, data-loss, money). These are objective; I recommend fixing them.
- **PART 2 — Design findings** (Apple-like polish). These are taste calls — accept or reject each independently.

> Nothing in the codebase was changed. This is findings-only. Proposed fixes are described, not applied.

## How this audit was run

1. **Multi-user simulation harness** (`scratch/audit_4.8/sim.py`) — boots the real Flask app threaded against an isolated SQLite DB, rate-limits off, and drives many concurrent simulated users through realistic workflows (create/edit/delete trips · days · expenses with splits · budgets · settlements · invites · feed shares · PDF · public shares · media). It asserts hard invariants; any 5xx or violated invariant is a bug. **Result: 44 invariants pass, 5 distinct bugs reproduced.**
2. **Four deep code reviews** (whole-file reads, not excerpts) across: trip write paths · money · social/feed · platform (pdf/public/auth/integrations). Findings in `scratch/audit_4.8/findings_*.md`.
3. **Live pixel design audit** — Playwright drove the real seeded app and screenshotted all 14 pages at desktop (1280×800) and mobile (375–390×812). Shots in `scratch/audit_4.8/shots/`.
4. **Browser E2E** — the project's own Playwright suite (non-visual) was run against an isolated DB to confirm golden paths in a real browser. **Result: 52 passed, 0 failures** across desktop + mobile (49 visual-regression snapshot tests excluded) — create trip, add companion/day/expense, mobile bottom-sheet modal, tab-nav, public share, photo flows, and axe-core a11y gates (home/profile/settings/friends) all green.

**Baseline:** the existing `pytest` suite is green (491 passed). This codebase has already survived 12 audit rounds (R1–R12); these findings are what those rounds **missed**.

### Verification tags

- **[VERIFIED-DB]** reproduced by a direct SQLite test.
- **[TRACED]** confirmed by following the exact code path (not yet runtime-reproduced).

## P0 — Critical (security / privacy / data-loss)

### SOCIAL-2 — Blocked user can repost the blocker's public post · [VERIFIED-SIM]

- **Where:** `src/routes/feed.py:815-901` (`repost_feed_post`), the `if not original['is_public']:` branch at ~`:868`.
- **What:** For a **public** trip the repost route skips the visibility/auth check ("anyone can repost — Twitter model") and there is **no block check anywhere** in the route. A user B whom A has blocked can `POST /api/feed/repost/<A's public post id>` and re-inject A's content into B's followers' feeds.
- **Harness proof:** A blocks B; B successfully reposts A's public post (`status: "reposted"`).
- **Fix:** Resolve the original author and reject when a block edge exists in either direction, at the top of `repost_feed_post`. Add the same guard to `toggle_feed_like` / `add_feed_comment` / `toggle_feed_bookmark`.

### SOCIAL-1 — Blocked user sees + engages the blocker's content via a third party's repost · [TRACED]

- **Where:** `src/feed_events.py:481-516` (`_build_friend_reposted_trip`), `:213-221` (`_visible_to_post_friends`), actor filter `:800-811`.
- **What:** Blocks remove A↔B from each other's actor pool, but a repost is a row owned by the **reposter**. If C (friend of both) reposts A's public share, the `repost_<id>` event enters B's feed (C isn't blocked) and the card embeds A's trip + lists A as `original_sharer` (id, name, picture). The engagement gate only checks friendship with C, so B can also like/comment. Block teardown (`blocks.py:163-172`) deletes B's reposts of A, never C's.
- **Fix:** In `_build_friend_reposted_trip` exclude reposts whose original author is on either side of a block edge with the viewer (mirror the owner-block subquery already at `feed_events.py:357-362`), and enforce it in `_visible_to_post_friends`.

### TRIP-1 — Media edit silently dropped + local edit wiped during the hydration window · [TRACED]

- **Where:** `frontend/static/js/src/api.ts:791-804` (`persistTripMedia` silent-drop), `:742-773` (`fetchTripMedia` overwrite), `:726` (`_mediaLoadedTrips` is in-memory only). Edit surfaces with no hydration gate: `pages/home/tripChecklistModal.ts:50-52`, `pages/home/dayDetailModal.ts:588-590`, `pages/home-mount/HeroMap.tsx:326-328`, `pages/todo/Todo.tsx:401-409`.
- **What:** `_mediaLoadedTrips` resets empty on every page load. `persistTripMedia` **silently returns without writing** when the trip isn't hydrated yet, and no edit surface blocks editing during the in-flight `fetchTripMedia`. After a cold load or trip-switch the user can add a checklist item / marked place / photo; the POST is dropped, then `fetchTripMedia` resolves and overwrites the local item with the server's old array. The item appears, then vanishes, and never reached the server.
- **Why it matters:** This is the exact data-loss class the R12 invariant exists to prevent — re-entering through a timing window rather than the write path. Trip-switching is common; the window is one network round-trip.
- **Fix:** Don't silently drop. Block/disable media-edit surfaces until `_mediaLoadedTrips.has(id)` (await `fetchTripMedia` on trip-open), or have `fetchTripMedia` **merge** instead of overwrite when local edits are pending, or queue the dropped write and flush on hydration.

## P1 — Broken feature / wrong result / security weakness

### DAY-1 — Duplicate `day_number` returns HTTP 500 (the intended 409 is dead code) · [VERIFIED-SIM] · NEW

- **Where:** `src/routes/days.py:153` — `if "idx_trip_days_trip_day_number" in str(exc):`.
- **What:** SQLite's `IntegrityError` message for that partial unique index is `UNIQUE constraint failed: trip_days.trip_id, trip_days.day_number` — it contains the **column** names, never the **index** name. So the substring test never matches, the handler re-`raise`s, and the request 500s instead of returning the intended graceful 409. Every duplicate `day_number` hits this: two tabs/planners adding the same day, or renumber-after-delete (see TRIP-2).
- **Harness proof:** posting a second day with an existing `day_number` → 500 `Internal server error`. Direct SQLite repro confirms the message lacks the index name. No existing test covers this path, which is how it survived 491 tests.
- **Fix:** Match the column-name substring (`trip_days.day_number`) or simply catch `sqlite3.IntegrityError` for this INSERT and return 409. (`retry_on_lock` only catches `OperationalError`, so it doesn't help here.)

### MONEY-1 — Partially-scoped budgets double-count spend (the "B6 fix" is incomplete) · [VERIFIED-SIM + VERIFIED-DB]

- **Where:** `src/routes/budgets.py:130-153` (`ON CONFLICT(id)` only) + `UNIQUE(user_id,trip_id,category_id,owner_name)` (`database.py:440`) + partial index `idx_budgets_user_trip_generic` (`database.py:887`, migration `3df6a8b0c2d4`) which only covers the **both-NULL** scope.
- **What:** SQLite treats `NULL` as distinct in UNIQUE, so the constraint never dedupes a scope with a NULL component. The partial index only plugs the all-NULL case. The two "one set, one NULL" shapes — `(category set, owner NULL)` and `(owner set, category NULL)` — can be duplicated with a fresh `id`. `spentForBudget` then counts the same expenses under each duplicate → inflated "spent"/"allocated" → fake overspend.
- **Proof:** harness created two `(category=food, owner=NULL)` budgets, both persisted; a direct SQLite test confirms cases B and C insert duplicates while A and D are blocked.
- **Fix:** Store a non-NULL sentinel (`''`) for unscoped category/owner so the base UNIQUE bites, or add two more partial unique indexes for the half-scoped shapes. Mirror the coercion in the `/api/sync` budget loop.

### MONEY-3 — A non-party trip member sees the already-paid debt as outstanding · [VERIFIED-SIM]

- **Where:** `src/routes/data.py:1065-1075` ships only settlements where the caller is `from`/`to`; the balance math (`pages/settlement/balances.ts:191,363`) reads **only** `STATE.settlements`, populated **only** from `/api/data` (`api.ts:404`). The dedicated `/api/settlements/<trip>` endpoint (which _does_ return the row to any member) is never used to populate balance state.
- **What:** Trip {Alice, Bob, Charlie}; Bob pays a shared expense; Bob settles Charlie. Alice's `/api/data` never includes the Bob→Charlie settlement, so Alice's balance page never subtracts it — Alice keeps seeing a debt that's already paid and a wrong suggested-payment graph, contradicting what Bob and Charlie see.
- **Harness proof:** member C receives the settlement via `/api/settlements/<trip>` (`in_endpoint=True`) but **not** via `/api/data` (`in_data=False`), and balances.ts only reads the latter.
- **Fix:** Ship all of a trip's settlements to all accepted members, **or** compute the simplified-debt graph server-side and ship per-person balances (keeping individual rows private).

### SOCIAL-3 — A trip turned private after sharing keeps leaking into followers' feeds · [VERIFIED-SIM]

- **Where:** `src/feed_events.py:449-478` & `:481-516` (share/repost builders have **no `is_public` filter**); privacy-flip writes at `data.py:313` and `trips.py:170`; engagement gate `:213-221` checks friendship only.
- **What:** Sharing auto-promotes a trip to public and writes a `feed_posts` row. Only delete / account-wipe / explicit-unshare scrub that row. Toggling `isPublic` back to false (normal Home toggle → `/api/sync`) leaves the row, so the share card keeps rendering the trip's name/country/caption in followers' feeds, and likes/comments keep working. (The full trip body is protected — only the card metadata leaks.)
- **Harness proof:** follower still sees the trip's share card after the owner sets it private.
- **Fix:** Delete the trip's `feed_posts` (cascading reposts) when `is_public` transitions 1→0, or add `AND is_public=1` to both builders and require public-or-member in the engagement gate.

### SOCIAL-4 — Soft-revoked achievements still appear (and stay likeable) in feeds for 30 days · [TRACED]

- **Where:** `src/feed_events.py:593-599` (`_build_achievement_unlocked`) and `:247-254` (`_visible_to_achievement_friends`) — neither filters `revoked_at IS NULL`.
- **What:** `list_user_achievements` correctly hides revoked badges (`achievements.py:842-845`), but the feed builder selects purely on `earned_at >= -30 days`. A user who _lost_ a badge (e.g. un-archived trips, dropping below the country threshold) still shows "earned 🌍 Globe Trotter" in followers' feeds, and friends can engage with the stale event.
- **Fix:** Add `AND ua.revoked_at IS NULL` to the builder SELECT and the visibility lookup.

### TRIP-2 — Renumber-after-delete 500s on the tombstoned day's unique slot · [VERIFIED-SIM]

- **Where:** unique index `idx_trip_days_trip_day_number` (`database.py:916-918`) lacks `deleted_at IS NULL`; `delete_day` soft-deletes (`days.py:217-221`); client renumbers survivors (`handlers.ts:272-325`).
- **What:** A deleted day is tombstoned but still occupies its `(trip_id, day_number)` slot in the index. Renumbering a survivor into the freed number collides with the tombstone → (because of DAY-1) a raw **500**. Net: a permanent gap in numbering and a failed renumber; the client also mislabels the resulting 409 as a "stale edit" toast for a purely local single-user delete.
- **Harness proof:** delete day #2, renumber day #3 → #2 → 500.
- **Fix:** Add `WHERE … AND deleted_at IS NULL` to the unique index (via migration) so tombstones free their slot; fix DAY-1 so the collision degrades to a clean 409; and stop labeling a UNIQUE-collision 409 as "stale edit".

### TRIP-4 — Concurrent media edits from two devices silently last-write-wins · [TRACED]

- **Where:** `src/routes/trips.py:678-685` — `update_trip_media` does a bare `UPDATE trips SET <col>=? WHERE id=?` with no `updated_at` check/bump; the frontend POSTs the entire array (`api.ts:798-803`).
- **What:** By design the media path is decoupled from the optimistic-concurrency token, and the client sends the full array with no merge. Device A adds photo X; device B (which never saw X) adds Y and POSTs its list without X → X is gone. Most likely on `markedPlaces` (high-frequency map toggle) and shared trips. This is the R12 read-modify-write class re-introduced on the media path.
- **Fix:** Gate the media write with a version token, or make it a server-side element-level merge (add/remove by id) instead of full-array replace.

### TRIP-3 — `/api/sync` still writes the 4 heavy media columns · [TRACED]

- **Where:** `src/routes/data.py:363-366` (active loop) and `:462-464` (archived loop).
- **What:** The invariant says media writes flow only through `POST /api/trips/<id>/media`. `/api/sync` still upserts `marked_places/documents/photos/checklist` (COALESCE-guarded, so absent keys are safe — but a payload that _includes_ `photos:[]` overwrites). The first-party client only sends `categories` today, so it's dormant, but the handler advertises the bulk path for "legacy clients" and "defensive re-syncs," and any such caller would clobber media — bypassing the entire isolation guarantee.
- **Fix:** Drop the 4 media columns from both `/api/sync` trip loops (mirror `upsert_trip`'s `None`). Makes the isolation structural rather than client-trust-dependent.

### PLAT-3 — Any logged-in user can read ANY user's private uploads/receipts · [TRACED]

- **Where:** `src/main.py:1223-1226` — the authenticated branch of `serve_upload` returns `send_from_directory` with no ownership/membership check.
- **What:** Protection relies solely on the unguessable filename. Anyone who once had access to a photo/document/**receipt** URL (a removed/declined trip member, or a URL harvested from a `/api/public-trip` or `/api/data` payload, browser history, or a log) keeps **permanent** read access — files are only physically deleted when the whole trip/day is deleted, never when a member leaves.
- **Fix:** Gate by `caller_id == owner_dir` OR accepted-membership of a trip referencing the file (at minimum for receipts).

### PLAT-2 — Session cookie can ship without `Secure` in production · [TRACED]

- **Where:** `src/auth.py:405,423` via `_is_secure_request` → `request.is_secure`; same detection underpins HSTS (`main.py:503`) and the share cookies (`main.py:1146`, `public.py:732`).
- **What:** `request.is_secure` is only true when ProxyFix translated a trusted `X-Forwarded-Proto: https`. If that header is ever absent, or the proxy topology differs from `GG_TRUSTED_PROXIES=1`, the 30-day session JWT cookie is minted **without** `Secure` and can leak over a plain-HTTP downgrade — defeating much of the point of the HttpOnly-cookie migration.
- **Fix:** In non-dev, set `secure=True` unconditionally; reserve auto-detect for local dev.

## P2 — Edge case / degraded / clarity

### MONEY-2 — Fully-scoped duplicate budget → unhandled IntegrityError → 500 · [TRACED]

`src/routes/budgets.py:130-153`. A second fully-scoped budget `(category set, owner set)` with a new `id` hits the base UNIQUE → `IntegrityError`, uncaught → 500 (the optimistic UI then shows a raw server-error toast). **Fix:** try/except `IntegrityError` → 409 with a clear message.

### MONEY-4 — Per-trip vs cross-trip balance use different split denominators · [TRACED]

`pages/settlement/balances.ts:179` (per-trip: roster = current ∪ attributed) vs `:324-336` (global: current companions only). A split-less expense involving someone no longer a current companion divides by different denominators in the two views → the same person's balance disagrees between the per-trip and cross-trip sheets. Live only for legacy/API split-less rows (the form always sends splits). **Fix:** derive the equal-split group identically in both.

### MONEY-5 — `simplifyDebts` €0.50 epsilon silently drops real sub-50-cent debts · [TRACED]

`pages/settlement/balances.ts:212-233` (`_ZERO_EPSILON_EUR = 0.5`) is applied to the **net balance**, not just FX residue. A genuine €0.40 debt is classified as neither creditor nor debtor → never appears in suggested payments and can't be settled in the UI; "All settled 🥂" can show with real imbalance. **Fix:** use a tight epsilon (~€0.005) for classification; handle FX residue with a per-trip sweep instead.

### MONEY-6 — Server never checks that splits sum to ~100 · [TRACED]

`src/validators.py:169-205` caps each split value to [0,100] but never checks the sum or key count. A crafted `splits={"a":100,"b":100}` is accepted; readers normalize by Σ so it silently becomes a 50/50 ratio rather than the percentages the model assumes. Latent contract violation + unbounded key-count fuzz vector. **Fix:** reject `|sum-100| > tol`; cap key count.

### MONEY-7 — Non-EUR settlement with crafted `euroValue:0` corrupts balance · [TRACED]

`src/routes/settlements.py:190-222` only rejects when `rate is None AND euro_value is None`; `euroValue:0` passes, `compute_euro_value` returns 0 (`0>=0`), and `balances.ts:103` does `euroValue || amount` → 0 is falsy → the raw **foreign** amount is treated as EUR (e.g. 5000 THB counted as €5000). Needs the cold-FX path + a crafted client. **Fix:** treat `euro_value<=0` as missing for non-EUR; use an explicit `!= null` fallback.

### MONEY-8 — Home currency forced to EUR for users outside the 17-entry static table · [TRACED]

`frontend/static/js/src/utils/currency.ts:33-40` gates `getHomeCurrency` on the static `CONVERSION_RATES` (17 codes) although the app supports ~41. A THB/EGP/TRY/PLN user has their display currency silently coerced to EUR everywhere (entry dropdowns were widened to the live cache, the home/display currency wasn't). **Fix:** gate on `hasRate(set)` (live ∪ static ∪ EUR).

### SOCIAL-6 — Comment block-filter and engagement counts are one-directional · [TRACED]

`src/routes/feed.py:1013-1023` (comment list filters only `blocker_id = caller`, not who-blocked-caller) and `:159-187` (`_attach_engagement_counts` does no block filtering). On a mutually-visible thread the blocked party still reads the blocker's comments, and counts leak interaction existence across a block. **Fix:** exclude either-side-of-block in both the comment-list and count queries.

### PLAT-1 — PDF "Total planned" sums mixed currencies as EUR · [TRACED]

`src/routes/pdf.py:1948-1962` accumulates raw `budgets.amount` across currencies and labels the sum EUR, even though per-row budgets print their real currency. A `USD 1000` + `GBP 500` trip prints "Total planned EUR 1500" — a wrong figure in a shareable artefact, next to the genuinely-EUR "Actual trip spend". **Fix:** EUR-normalise before summing, or show per-currency subtotals.

### PLAT-4 — Anonymous cover-image fetch ignores `is_archived` · [TRACED]

`src/main.py:1260-1268` serves a cover if any trip has `is_public=1 OR share_token IS NOT NULL`, but never checks `is_archived` (unlike `fetch_share_payload`, `public.py:514`). After an owner archives a shared trip (supposed to stop serving it), the cover PNG stays anonymously fetchable. **Fix:** add `AND COALESCE(is_archived,0)=0`.

## By-design note (confirm against product intent)

- **Turning a trip private does NOT invalidate existing share links** (`is_public` and `share_token` are independent; `feed.py:806` leaves the token). # PART 2 — Design findings (Apple-like polish)

> These are **taste calls** — accept or reject each independently. Screens: `scratch/audit_4.8/shots/` (14 pages × desktop + mobile).

**The core tension:** the app currently reads **colorful / playful** — rainbow per-section accent colors, gradient titles, emoji used as UI icons. The stated goal is **sharp, minimal, Apple-like**. Most items below ladder up to _unify and restrain the visual language_.

## Systemic (highest leverage)

- **DSGN-1 — Page-title colors span the whole rainbow.** Feed = indigo, Expenses/Insights/Search/To-do/AI = blue→purple gradient, Settlements/Budgets = **gold**, Settings ("System Control") = **green**, Friends/Collections = blue. Even the two cards inside Settings use different accents (blue vs orange). Apple-like is restrained. **Proposed:** near-black page titles + one brand accent; reserve color for status (success/warn/danger) only.
- **DSGN-2 — Emoji used as UI icons everywhere** (🔍 📣 🗺️ 📍 💰 ⚖️ 📌 📅 📋 📚). Renders inconsistently per-OS and reads informal. **Proposed:** one monochrome line-icon set (SF-Symbols-style), sized to the text.
- **DSGN-3 — Oversized mobile hero titles.** "Feed" / "Expenses" / "Search" / "Plan with AI" / "Your network" + subtitle consume ~25–30% of the mobile fold before any content. **Proposed:** smaller type scale + tighter vertical rhythm on mobile.
- **DSGN-4 — Gradient-heavy.** Blue→purple gradient titles + a multicolor wordmark. Flat, confident color is sharper. **Proposed:** flat title color; keep at most one subtle brand gradient.
- **DSGN-5 — Bottom tab bar is icon-only and ambiguous.** Four unlabeled icons (Home / ✓ / ✦ AI / 💳 expenses); ✦ and 💳 aren't obvious, and the active state is only a faint blue rounded highlight. **Proposed:** add text labels, a clearer active state, and a visually distinct primary action (Apple HIG tab bars are labeled).

## Layout / mobile defects

- **DSGN-6 — Segmented controls truncate on mobile.** Friends shows "Follow… 0 / Followi… 0" (Followers vs Following are indistinguishable); Expenses "One at a / time" wraps to two lines; a budget label truncates to "… · AI…". **Proposed:** shorter labels / size-to-content / smaller text.
- **DSGN-7 — Map hero dominates Home and its failure state is ugly.** Without a Maps key it's a large gray "Oops! Something went wrong" block filling the mobile fold; even the success hero is tall and pushes the day plan below the fold on mobile. **Proposed:** a graceful map-unavailable state + a shorter hero on small screens.
- **DSGN-8 — Stray "/" glyph** floats in the Budgets OVERALL card between SPENT and ALLOCATED.
- **DSGN-9 — Settlements shows two different "owed" numbers on one screen** — headline "+ MOST OWED Alex +$565.09" vs "Trip balances · Alex +$512.70" (differ by the $52.40 already settled), unlabeled. Reads as a contradiction. **Proposed:** label gross-vs-net or show one number. (Relates to MONEY-3/5.)
- **DSGN-10 — Double heading on Insights** — the "Expenses" page title and a second giant "Insights" title stack, wasting vertical space.

## Consistency / polish

- **DSGN-11 — Inconsistent page-naming voice** — "System Control", "Your network", "Your to-do list" vs literal "Settlements"/"Budgets"/"Feed". Pick one voice.
- **DSGN-12 — Google Sign-In button renders in Portuguese** ("Iniciar sessão com o Google") while the UI is English (the GIS widget follows its own locale). **Proposed:** pass the app locale to Google Identity Services or use a custom-styled button.
- **DSGN-13 — Insights currency selector is independent of home currency** → different units across pages at once (€ on Insights vs $ on Budgets/Settlements in testing). **Proposed:** default Insights to home currency, or label clearly.
- **DSGN-14 — Two ambiguous circular badges** sit top-right on desktop (a green-check ring + a red/pink one) next to the trip selector; meaning unclear. **Proposed:** label / tooltip / remove.
- **DSGN-15 — Desktop under-uses horizontal space** on Feed/Settlements/Friends (narrow centered column, big empty gutters). **Proposed:** wider or multi-column desktop layouts.
- **DSGN-16 — Wordy helper copy** (Friends intro paragraph, To-do empty state, AI quota) — tighten for a more confident, cleaner feel.
- **DSGN-17 — Verify dark-mode parity** — all captures were light; confirm a polished dark theme exists across every page.

## What already looks good (keep)

Clean white card system with soft shadows and generous radius · strong empty states (To-do, Search, Collections) with clear CTAs · the avatar treatment (gradient ring + camera affordance) · iOS-style segmented pills · a consistent top app bar · generous whitespace · an attractive, value-communicating login wall.
